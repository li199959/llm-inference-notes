## 1.1 训练与推理：根本不同的目标

当训练大语言模型时，目标是在海量数据集上最小化损失函数。系统以大批量处理数据，在许多 GPU 之间累积梯度，执行反向传播，并更新数十亿个参数。吞吐量是最重要的指标：我们希望在整个训练过程中摊销后，每秒处理尽可能多的 token。单次前向传播的延迟基本无关紧要；真正重要的是以天或周为单位衡量的总体训练时间。

推理则像是训练的镜像。在这里，模型参数被冻结。任务是计算一次前向传播，也就是把输入提示转换为预测输出，并且要尽可能快、尽可能低成本地为每个单独请求完成这一过程；这些请求往往来自正在等待响应的真实用户。

**定义：推理**

推理是指在新的、未见过的输入数据上运行已经训练好的神经网络模型，以产生预测或输出，而不更新模型的参数。

推理的目标在三个关键维度上与训练明显不同：

- **延迟优先于吞吐量**：等待响应的用户会直接感受到延迟。200 ms 的响应让人感觉瞬时完成；3 秒的响应则让人感觉缓慢。
- **内存效率**：在推理时，我们通常不需要存储优化器状态或梯度缓冲区，但确实需要管理用于自回归生成的 KV Cache（键值缓存）。
- **每请求成本**：在生产环境中，你要服务数百万个请求。每个请求的成本会直接决定盈利能力。

## 1.2 Token 生成循环详解

现代 LLM 是自回归的：它们一次生成一个 token，每个新 token 都以此前生成的所有 token 为条件。这构成了 LLM 推理的基本循环。

**自回归生成循环**

1. 接收一个提示词（输入 token 序列）。
2. 对整个提示词并行运行一次 prefill（预填充）前向传播，为所有输入 token 计算注意力的 key 和 value，并将它们缓存起来。
3. 自回归解码：在每一步中，将上一个生成的 token 作为输入，基于 KV Cache 计算注意力，并采样下一个 token。
4. 重复第 3 步，直到生成停止 token 或达到最大长度。

prefill 阶段受计算限制：我们并行处理许多 token，充分利用 GPU 的张量核心。然而，decode 阶段从根本上受内存带宽限制：我们每次只处理一个 token，但每一步都必须从 GPU 内存中加载整个 KV Cache。正是这种不对称性，驱动了推理系统中的大量复杂性。

## 1.3 延迟、吞吐量与成本：铁三角

推理优化处在一个三难困境之中。改进其中一个维度，往往要以牺牲另一个维度为代价：

| 指标 | 定义 | 如何改进 |
| --- | --- | --- |
| 延迟 | 从请求发出到完成的时间。关键子指标包括 TTFT（Time to First Token，首 token 时间）和 TPOT（Time Per Output Token，每输出 token 时间）。 | 缩小模型规模，谨慎增大 batch size，使用 speculative decoding（推测解码），优化 KV Cache。 |
| 吞吐量 | 所有并发用户合计每秒生成的 token 数。 | 增大 batch size，使用 continuous batching（连续批处理），使用 tensor parallelism（张量并行）。 |
| 成本 | 每生成 100 万个 token 的美元成本。 | 量化模型，使用高效的服务框架，优化硬件利用率。 |

铁三角会以具体方式体现出来：最大化吞吐量需要较大的 batch size，但大 batch 会增加每个请求的延迟。最小化延迟通常需要单独服务请求，这会浪费 GPU 利用率并增加成本。推理工程的艺术，就是根据你的具体应用需求，聪明地驾驭这些权衡。

## 1.4 内存带宽：真正的瓶颈

一个常见误解是推理受计算限制，也就是我们需要更多 FLOPS。实际上，对于自回归 LLM 解码，瓶颈几乎总是内存带宽：也就是我们能以多快的速度把模型权重从 GPU 内存（HBM）加载到 GPU 的计算核心。

考虑一个以 16 位精度存储的 70 亿参数模型。模型权重大约占用：

7 × 10^9 × 2 bytes = 14 GB

一块 NVIDIA A100 GPU 提供约 2 TB/s 的 HBM 带宽。要加载所有权重一次（一个 decode 步骤），延迟的理论下界是：

14 GB

2000 GB/s = 7 ms

这还没有计入任何计算。如果我们以 batch size 1 运行，就会花费这段时间加载权重，只为生成一个 token。若 batch size 为 64，我们就能把同一次权重加载摊销到 64 个同时处理的请求上，在几乎不改变延迟的情况下显著提高吞吐量。

## 1.5 Roofline 模型

Roofline 模型为理解某个给定操作究竟受计算限制还是受内存限制提供了一个清晰框架。对于任意操作：

Achieved Performance (FLOP/s) = min (Peak FLOP/s, Arithmetic Intensity × Peak Bandwidth)

其中 Arithmetic Intensity（算术强度）以每访问 1 字节内存所执行的 FLOP 数来衡量。

**实用 Roofline 洞察**

矩阵-向量乘法（batch size 为 1 时 decode 步骤的核心）具有较低的算术强度（约 1-2 FLOP/byte），因此稳稳处在受内存限制的区域。大规模矩阵-矩阵乘法（长提示词的 prefill）具有较高的算术强度，并接近计算上限。这就是为什么 prefill 和 decode 的行为如此不同，也解释了为什么它们会受益于不同的优化。

## 面试准备：模拟问答

**Q1.1. LLM 推理中的 prefill 和 decode 有什么区别，为什么这对系统设计很重要？**

**回答：**prefill 阶段在一次并行前向传播中处理整个输入提示词，因此它受计算限制，因为我们会将大型 token 矩阵与权重矩阵相乘。decode 阶段以自回归方式一次生成一个 token，因此它受内存带宽限制，因为每个单 token 步骤都要加载全部模型权重。这一区分对系统设计极其重要：prefill 和 decode 具有非常不同的最佳 batch size、GPU 利用率特征和延迟敏感性。像 Splitwise 和 Mooncake 这样的现代系统会将 prefill 和 decode 在物理上分离到不同的 GPU 池中，使二者可以独立扩缩容和优化。

**Q1.2. 请解释推理的铁三角。对于一个延迟敏感型聊天机器人应用，你会如何做优化决策？**

**回答：**推理铁三角描述的是延迟、吞吐量和成本之间的张力；通常改进其中一项会损害另一项。对于延迟敏感型聊天机器人，我会优先考虑 TTFT 和 TPOT。具体而言：（1）使用更小或经过量化的模型，以减少每步计算量。（2）保持较小的 batch size，或使用带有严格 SLO 感知调度的 continuous batching。（3）应用 speculative decoding，以便每步生成多个 token。（4）使用 prefix caching（前缀缓存），跳过对重复系统提示词的 KV Cache 重新计算。我会接受更高的每 token 成本，将其作为低延迟的权衡代价。

**Q1.3. 为什么在 LLM 解码中，瓶颈通常是内存带宽，而不是原始计算能力（FLOP/s）？**

**回答：**在小 batch size 的自回归解码过程中，GPU 每生成一个 token 都必须从 HBM 加载整个模型权重。一个 7B 参数的 FP16 模型为 14 GB；即使在带宽为 2 TB/s 的 A100 上，每一步也大约需要 7 ms。与此同时，单个 token 上的实际计算量与 GPU 张量核心能够处理的规模相比非常小。矩阵-向量乘法（关键的 decode 操作）的算术强度大约是每字节 1-2 FLOP，远低于达到受计算限制所需的约 100 FLOP/byte 阈值。只有通过增大 batch size，我们才能摊销权重加载成本，并接近受计算限制的行为。

**Q1.4. 你如何衡量和报告推理性能？在生产服务系统中，你会埋点哪些指标？**

**回答：**我会跟踪：TTFT（time to first token，首 token 时间，对用户感知响应速度至关重要）、TPOT（time per output token，每输出 token 时间，决定流式输出的平滑程度）、throughput（所有用户合计的 tokens/second）、GPU 利用率和内存利用率、goodput（满足 SLO 的请求比例）、queue depth（队列深度，用于检测过载），以及 cost per million tokens（每百万 token 成本）。我会设置百分位数仪表盘（p50、p95、p99），而不是只看平均值，因为尾延迟才是破坏用户体验的因素。我还会分别跟踪 prefill 延迟和 decode 延迟，以便独立诊断瓶颈。

**Q1.5. 一位同事认为，由于 FLOP/s 的增长速度持续快于内存带宽，推理很快就会变成受计算限制。你如何回应？**

**回答：**这是经典的 bandwidth wall（带宽墙）论点；我会部分同意这个观察，但不同意它的结论。确实，近几代 GPU 的计算 FLOP/s 增长速度快于 HBM 带宽。然而，单 batch decode 的算术强度非常低（1-2 FLOP/byte），因此即使 FLOP/s 大幅提升，除非 batch size 显著增大或模型架构发生变化，我们仍然会受内存限制。此外，量化等技术会有意用计算换取更少的内存流量，从而让我们继续处在受内存限制的区域。前进方向是提升每加载 1 字节所对应算术强度的架构和系统，比如 GQA、MLA 和高效的 KV Cache，而不是单纯寄希望于更快的计算。
